1 # Changes of Note
2 - PCL `--visualize` capability of the `pdal` command line application has been removed.
3 - `writer.derivative` has been removed. Use [gdaldem](http://www.gdal.org/gdaldem.html) for faster and more featureful equivalent functionality.
4 - GeoTIFF and Proj.4 are now required dependencies.
5 - `writers.p2g` has been removed. It was replaced by `writers.gdal` in 1.4, but the P2G writer was essentially unmaintained and we will be using the GDAL one going forward.
6 - `filters.attribute` was split into `filters.assign` and `filters.overlay` to separate their functionalities
7 - `filters.pmf` and `filters.outlier` have dropped the `classify` and `extract` options. They now only classify points and leave it downstream filters to ignore/extract classifications as needed.
8 - `filters.outlier` has changed the default classification for noise points from `18` to `7` to match the LAS classification code for "Low point (noise)".

10 # Enhancements
11 
12 - `pdal pipeline` now supports a `--stream` option which will default to one-at-a-time or chunk-at-a-time point processing when all stages in the pipeline support it. You can use this option to control memory consumption -- for example when interpolating a very large file with `writers.gdal`
13 - `filters.crop` was enhanced to support transformed filter polygons, streaming, and radius cropping.
14 - `readers.greyhound` updated to support greyhound.io 1.0 release, with the most significant enhancement being support for passing downstream JSON filters.
15 - `user_data` JSON object can be applied to any PDAL pipeline object and it will be carried through processing. You can use this mechanism for carrying your own information in PDAL pipelines without having to sidecar data. #1427
16 - `writers.las` now can write `pdal_metadata` and `pdal_pipeline` VLRs for processing history tracking. #1509 #1525
17 - `metadata`, `schema`, and `spatialreference` objects added to global module for `filters.programmable` and `filters.predicate` Python filters.
18 - `pdalargs` option for `filters.programmable` and `filters.predicate` allow you to pass in a JSON dictionary to your Python module for override or modification of your script
19 - Stage tags can be used in pipeline override scenarios
20 - User-settable VLRs in `writers.las` #1542
21 - `filters.sort` now supports descending order and uses `std::stable_sort` #1530 (Thanks to new contributor @wrenoud )
22 - `pdal tindex` will now use data bounds if `filters.hexbin` cannot be loaded for boundaries #1533
23 - `filters.pmf` and `filters.smrf` improved performance #1531 and #1541
24 - `filters.assign` now supports [Range](https://www.pdal.io/stages/filters.range.html)-based filters
25 - `filters.outlier` now accepts a user-specified `class` to override the default value of `7` for points deemed outliers. #1545
26 - `filters.pmf` and `filters.smrf` now accept a [Range](https://www.pdal.io/stages/ranges.html#ranges) via the `ignore` option to specify values that should be excluded from ground segmentation. #1545
27 - `filters.pmf` and `filters.smrf` now consider only last returns (when return information is available) as the default behavior. The `last` option can be set to `false` to consider all returns. #1545

35 ## New filters
36 
37 - `filters.cluster` - Perform Euclidean cluster extraction, and label each point by its cluster ID. By @chambbj.
38 - `filters.groupby` - Split incoming PointView into individual PointViews categorically, e.g., by Classification. By @chambbj.
39 - `filters.locate` - Locate and return the point with the minimum or maximum value for a given dimension. By @chambbj.
40 - `filters.emf` - Extended Local Maximum filter. By @chambbj.
