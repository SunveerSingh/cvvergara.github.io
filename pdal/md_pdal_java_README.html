PDAL Java bindings

Java bindings to use PDAL on JVM (supports PDAL >= 1.4).

Using PDAL JNI with SBT

1 // pdal is published to maven central, but you can use following repos in addition
2 resolvers ++= Seq(
3  Resolver.sonatypeRepo("releases"),
4  Resolver.sonatypeRepo("snapshots") // for snaphots
5 )
6 
7 libraryDependencies ++= Seq(
8  "io.pdal" %% "pdal" % "1.6.0"
9 )

It's required to have native JNI binary in java.library.path:

1 // Mac OS X example with manual JNI installation
2 // Though it's strongly recommended to use WITH_PDAL_JNI during PDAL build
3 // cp -f native/target/resource_managed/main/native/x86_64-darwin/libpdaljni.1.4.dylib /usr/local/lib/libpdaljni.1.4.dylib
4 // place built binary into /usr/local/lib, and pass java.library.path to your JVM
5 javaOptions += "-Djava.library.path=/usr/local/lib"

PDAL-Scala

Scala API to build pipeline expressions instead of writing a raw JSON.

1 libraryDependencies ++= Seq(
2  "io.pdal" %% "pdal-scala" % "1.6.0"
3 )

Scala API covers PDAL 1.6.0 but is compatible with PDAL >= 1.4, to use any custom DSL that is not covered by the current Scala API you can use RawExpr type to build Pipeline Expression.

Code examples

1 // To construct the expected json
2 val expected =
3  """
4  |{
5  | "pipeline" : [
6  | {
7  | "filename" : "/path/to/las",
8  | "type" : "readers.las"
9  | },
10  | {
11  | "type" : "filters.crop"
12  | },
13  | {
14  | "filename" : "/path/to/new/las",
15  | "type" : "writers.las"
16  | }
17  | ]
18  |}
19  """.stripMargin
20 
21 // The same, but using scala DSL
22 val pc: PipelineConstructor = LasRead("/path/to/las") ~ CropFilter() ~ LasWrite("/path/to/new/las")
23 
24 // The same, but using RawExpr, to support not implemented PDAL Pipeline API features
25 // RawExpr accepts a circe.Json type, which can be a json object of any desired complexity
26 val pcWithRawExpr = LasRead("/path/to/las") ~ RawExpr(Map("type" -> "filters.crop").asJson) ~ LasWrite("/path/to/new/las")

How to compile

Development purposes (including binaries):

  1. Install PDAL (using brew / package managers (unix) / build from sources / etc) without -DWITH_PDAL_JNI=ON flag
  2. Build native libs ./sbt native/nativeCompile (optionally, binaries would be built during tests run)
  3. Run ./sbt core/test to run PDAL tests

Only Java development purposes:

  1. Provide $LD_LIBRARY_PATH or $DYLD_LIBRARY_PATH
  2. If you don't want to provide global variable you can pass -Djava.library.path=<path> into sbt: ./sbt -Djava.library.path=<path>
  3. Set PDAL_DEPEND_ON_NATIVE=false (to disable native project build)
  4. Run PDAL_DEPEND_ON_NATIVE=false ./sbt

Finally the possible command to launch and build PDAL JNI bindings could be:

1 # Including binaries build
2 # WARN: PDAL should be built without `-DWITH_PDAL_JNI=ON` flag
3 ./sbt
1 # Java side development without binaries build
2 # WARN: PDAL should be built with `-DWITH_PDAL_JNI=ON` flag
3 PDAL_DEPEND_ON_NATIVE=false ./sbt -Djava.library.path=<path>

Possible issues and solutions

  1. In case of not installed as global PDAL change this line to:
1 set(CMAKE_CXX_FLAGS "$ENV{PDAL_LD_FLAGS} $ENV{PDAL_CXX_FLAGS} -std=c++11")

In this case sbt launch would be the following:

1 PDAL_LD_FLAGS=`pdal-config --libs` PDAL_CXX_FLAGS=`pdal-config --includes` ./sbt
  1. Sometimes can happen a bad dynamic linking issue (somehow spoiled environment), the quick workaround would be to replace this line to:
1 set(CMAKE_CXX_FLAGS "-L<path to dynamic libs> -std=c++11")
